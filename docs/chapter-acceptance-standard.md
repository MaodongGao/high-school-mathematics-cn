# 章节自主整理与验收标准

本标准把“数列”一章反复校订后形成的要求固化为未来章节的默认工作流。目标是让代理可以持续推进完整章节，而不是交付一个能编译但内容不完整的初稿。

校准时直接参考已经验收的 `02-sequences.tex`、`math-11-example-audit.md` 与
`math-11-review-notes.md`。涉及几何图时，以 `01-space-geometry.tex` 中“图形与解法一致”的做法为基准。

## 一、开始前：先建立覆盖地图

在写正文之前，代理必须：

1. 阅读来源 PDF 的全部目标页，确认主题边界；
2. 从 `sources/example-audit-template.md` 复制逐页核对表；
3. 登记每一道例题、每个小问、同题多解、重复练习、重要旁注、图形和二级结论；
4. 在 `sources/catalog.csv` 中把状态设为 `transcribing`，不得直接设为 `complete`；
5. 从 `sources/review-notes-template.md` 建立 review 文档。

逐页表是正文的覆盖地图，不是完成后补写的总结。原稿中一题三问必须登记三问；同一道题在后文重复出现，也必须另有一行说明“合并整理”，不能因重复而消失。

## 二、正文质量标准

### 内容忠实

- 原稿定义、推导节奏、代表题、难题和有启发性的长解法应被保留。
- 同题多解若体现不同结构，必须同时保留。
- 不得把“信息密度”误解为只给结论或只留最短解。
- 代理添加的题必须标为“补充”，并与原稿难度和方法价值匹配。
- 原稿错误要修正，但必须在正文“辨析”或 review 文档中留痕。

### 可理解性与结构

- 定义、定理、推论、例题之间必须有完整的理解链。
- 对难题有迁移价值的二级结论使用绿色 `corollarybox`，不能埋在普通段落。
- 内容顺序按依赖关系和学习节奏组织；由例题归纳的结论放在例题后。
- 例题解答应写出关键转化为何成立，不能从题设跳到答案。

### 图形

- 几何例题必须有与实际解法一致的 TikZ 图。
- 图中必须出现解答使用的辅助线、射影、截面、角和关键标签。
- 难以仅靠文字重建的几何推论也应配图。
- 图不是装饰：错误或无关的图比没有图更糟。

## 三、数学复核

每一章至少进行一次独立验算，不直接相信原稿或第一次转录。逐项检查：

- 初值、通项和递推式能否互相代回；
- 等号与严格不等号是否正确；
- 起始下标、边界项和定义域是否完整；
- 极值发生在第几项，是否存在并列；
- 放缩方向、求和端点和裂项后的常数；
- 几何图中的点、线、角与文字推导是否一致；
- 同题多解是否得到同一答案；
- 原稿结论若错误，正文是否已修正并留下 review 记录。

## 四、覆盖复核

正文完成后，重新从来源第一页走到最后一页：

- 每个审计行都能指向正文中的明确位置，或有具体合并/不收录原因；
- 不允许使用“篇幅原因”“不重要”等空泛理由跳过题目；
- 难题、代表题和独特方法不得有意不收录；
- `待收录` 和 `暂缓` 必须继续处理，除非确实需要用户辨认；
- 所有需要用户确认的内容集中写入 review 文档，不能散落在聊天记录中。

## 五、构建与 PDF 复核

必须执行：

```bash
make editorial-check
PATH=/Library/TeX/texbin:$PATH make verify
make render-review
```

代理需要查看 contact sheet，并以原始尺寸查看所有新增、跨页、含图或公式密集的页面。检查：

- 裁切、重叠、过满或异常留白；
- 例题框跨页是否自然；
- 三种语义颜色是否清楚区分；
- 图中文字是否可读，辅助线是否正确；
- 页眉、页脚、章节标题和页码；
- 最终 PDF 是否出现 `qquad`、`cdots`、`frac` 等漏反斜杠的命令名。

## 六、`complete` 的硬条件

只有同时满足以下条件，才可把来源状态改为 `complete`：

- 逐页审计表存在且无 `待收录`、`暂缓`；
- 所有合并或有意不收录都有具体理由；
- review 文档存在，且“未解决”明确写为“无”；
- 原稿例题、各小问和重要多解已经对账；
- 数学独立验算完成；
- 几何图要求满足；
- `make verify` 通过；
- 整章 PDF 已进行视觉检查；
- 最终交付说明列出合并项、原稿错误修正和代理新增内容。

若任一条件未满足，保持 `review`，继续完成所有不受阻的项目。不要因为已经投入很多工作、页面很多或编译成功而提前宣布完成。

## 七、一次性交付说明

为减少用户反复追问，最终交付应一次说明：

- 完成的来源页码和目标章节；
- 原稿项目总数，以及已收录、合并、有意不收录的数量；
- 所有合并项的去向；
- 发现并修正的原稿错误；
- 代理新增的题、推论或图；
- 仍未解决的问题；若没有，明确写“无”；
- `make verify` 与整章视觉检查结果；
- 最终 PDF、审计表和 review 文档的路径。

长时间自主运行时，不要在每个局部歧义处停下。先记录到 review 文档，继续完成其他页面，最后只把真正需要用户决定的项目集中列出。
